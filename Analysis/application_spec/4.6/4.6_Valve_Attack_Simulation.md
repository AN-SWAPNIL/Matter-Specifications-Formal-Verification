## Executive Summary

This document presents attack simulations for the three TRUE VULNERABILITIES identified in the Valve Configuration and Control cluster. These vulnerabilities exist due to specification gaps rather than implementation flaws - **all 20 defined security properties HOLD** when the specification is correctly implemented.

| Vulnerability | Severity | Exploitability | Physical Impact |
|---------------|----------|----------------|-----------------|
| VULN_001: Replay Attack | CRITICAL | Medium | Gas leak, flooding |
| VULN_002: Indefinite Open | CRITICAL | Low | Flooding, gas accumulation |
| VULN_003: No Fail-Safe Close | CRITICAL | Medium | Continued leak on fault |

```

│ "If the device has registered a fault that causes it to       │
│  prevent the valve to perform the requested action, the       │
│  command SHALL be ignored and a FailureDueToFault status      │
│  SHALL be returned."                                          │

SPEC DOES NOT SAY:
┌────────────────────────────────────────────────────────────────┐
│ ✗ "On Leaking fault, valve SHALL transition to Closed"        │
│ ✗ "Fail-safe position SHALL be configured per application"    │
│ ✗ "Gas valves SHALL close on any fault"                       │
│ ✗ "Safety-critical valves SHALL have watchdog close"          │
└────────────────────────────────────────────────────────────────┘

CONTRAST WITH GOOD PRACTICE:
┌────────────────────────────────────────────────────────────────┐
│ Gas appliance standards typically require:                     │
│ - Fail-safe closed position on loss of power                  │
```

## Conclusion

The Matter Valve Configuration and Control cluster has **sound security properties when correctly implemented** - all 20 defined properties HOLD according to spec analysis. However, three critical specification gaps create real-world vulnerabilities:

1. **Replay attacks** are possible because Timed Interaction is not required
2. **Indefinite open states** are permitted by null duration values
3. **No fail-safe close** means faulted valves may remain open during active leaks

These gaps should be addressed in future spec revisions, and implementers should exceed spec minimums for safety-critical deployments.

## Cross-Verification Evidence

### Verification Methodology

All three vulnerabilities were verified against:
1. **Application Spec** - Section 4.6 (within section) and Section 5.2 Door Lock (comparison)
2. **Core Spec** - Section 7.6.10 Timed Interaction, Section 11.10.7.2 General Commissioning

### VULN_001: Timed Interaction Gap - CONFIRMED

| Cluster | Command | Access | Timed Required? |
|---------|---------|--------|-----------------|
| **Valve (Section 4.6, Page 392)** | Open | O | **NO** |
| **Valve (Section 4.6, Page 392)** | Close | O | **NO** |
| **Door Lock (Section 5.2, Page 430)** | LockDoor | O T | **YES** |
| **Door Lock (Section 5.2, Page 430)** | UnlockDoor | O T | **YES** |

**Core Spec Reference (Page 471, Section 7.6.10):**
> "Timed interactions are used to limit the amount of time an action message is valid... used to prevent a timing attack on the system. For example, a malicious attacker could perform an 'intercept, interfere, and replay' procedure..."

**Verdict:** Matter protocol recognizes need for Timed Interaction on safety-critical physical actuators (Door Lock), but Valve cluster omits it. **VULNERABILITY CONFIRMED.**

### VULN_002: Indefinite Duration - CONFIRMED

**Application Spec Reference (Page 390, Section 4.6.7.1):**
> "A value of null SHALL indicate the duration is not set, meaning that the valve will remain open until closed by the user or some other automation."

**Core Spec Check:** No maximum duration requirement found for physical actuators.

**Verdict:** Spec explicitly permits indefinite open duration with no safety limit. **VULNERABILITY CONFIRMED.**

### VULN_003: No Fail-Safe Close - CONFIRMED

**Application Spec Reference (Page 394, Section 4.6.8.2.1):**
> "If the device has registered a fault that causes it to prevent the valve to perform the requested action, the command SHALL be ignored and a FailureDueToFault status SHALL be returned."

**Observation:** Fault handling only BLOCKS commands. No requirement to transition to fail-safe (closed) position.

**Core Spec Check:** "Fail-safe" in core_spec refers to commissioning fail-safe timer (Section 11.10.7.2), NOT physical actuator safety behavior.

**Verdict:** Neither application_spec nor core_spec requires automatic fail-safe close on fault. **VULNERABILITY CONFIRMED.**
