## 3. Defense-in-Depth Security Layers

The Account Login cluster implements 7 independent security layers:

### Layer 1: Admin Privilege Requirement (PRIMARY DEFENSE)

**Commands**: GetSetupPIN, Login

**Specification**: Access 'A' in Commands table

**Impact**: Attackers must be **trusted administrators** on the fabric to even attempt these commands. This single requirement eliminates the vast majority of potential attackers.

**Core Spec Enforcement**: Section 6.6.6 Access Control Privilege Granting Algorithm

### Layer 2: Timed Interaction Requirement

**Commands**: All (GetSetupPIN, Login, Logout)

**Specification**: Access 'T' in Commands table

**Impact**:

- Prevents replay attacks
- Commands expire if not executed within timeout
- Requires explicit time-bounded session

**Core Spec Enforcement**: Section 8.8.2.3 - NEEDS_TIMED_INTERACTION status code

### Layer 3: Fabric Scoping

**Commands**: All

**Specification**: Access 'F' in Commands table

**Impact**: Cross-fabric attacks are architecturally impossible. Commands can only be executed by nodes belonging to the same fabric as the Content App.

**Core Spec Enforcement**: Section 6.6.6.4 - UNSUPPORTED_ACCESS for fabric mismatch

### Layer 4: Rate Limiting

**Commands**: GetSetupPIN, Login

**Specification** (Page 534):

> "SHALL implement rate limiting to prevent brute force attack attempts. No more than 10 unique requests in a 10 minute period SHALL be allowed for either command..."

**Normative Language**: SHALL (mandatory)

**Impact**: Limits attack throughput to 525,600 attempts/year, making brute force infeasible even against weak PINs.

### Layer 5: TempAccountIdentifier Requirements

**Property 1**: Minimum length 16 characters (SHALL)

**Property 2**: Maximum validity 10 minutes (SHALL)

**Impact**:

- 16+ character minimum prevents enumeration
- 10-minute validity limits attack window
- Combined with rate limiting: max 10 guesses per TempId lifetime

### Layer 6: SetupPIN Requirements

**Specification** (Page 535):

> "SHALL... be at least 8 characters in length"

**Impact**: 8+ character minimum provides strong entropy against brute force even with rate-limited guessing.

### Layer 7: CRITICAL Priority LoggedOut Event

**Specification**: Events table shows Priority = CRITICAL

**Impact**: LoggedOut events cannot be filtered or deprioritized, ensuring reliable session termination signaling.

## 4. Intentional Specification Gaps

Three specification gaps were identified and classified as **intentional design decisions** with **no security impact**:

### GAP_001: External Account Verification

**Spec Text**: "The account match is determined by the Content App using a method which is outside the scope of this specification and will typically involve a central service..."

**Classification**: `INTENTIONAL_SCOPE_LIMITATION`

**Rationale**: Matter correctly defines protocol boundaries. External service security is the Content App provider's responsibility. This is standard practice for application-layer protocols.

### GAP_002: Advisory Aggressive Timeout

**Spec Text**: "Implementations... should impose aggressive time outs..."

**Classification**: `INTENTIONAL_NORMATIVE_LANGUAGE`

**Rationale**: Lowercase "should" is advisory per RFC 2119. Security is not compromised because:

- TempId 10-minute validity is mandatory (SHALL)
- Rate limiting is mandatory (SHALL)
- The advisory is for UX optimization, not security

### GAP_003: Conditional Account Activation

**Spec Text**: "Content App MAY make that user account active."

**Classification**: `INTENTIONAL_IMPLEMENTATION_FLEXIBILITY`

**Rationale**: MAY allows business logic flexibility (parental controls, subscription checks, etc.) without impacting authentication security. Authorization is fully enforced.

## 7. Mitigation Recommendations

### 7.1 Implementation Requirements (Mandatory)

These must be implemented exactly as specified with SHALL language:

1. **Admin privilege enforcement** for GetSetupPIN and Login commands
2. **Timed Interaction enforcement** for all commands
3. **Rate limiting**: No more than 10 requests per 10 minutes
4. **TempAccountIdentifier**: Minimum 16 characters, max 10-minute validity
5. **SetupPIN**: Minimum 8 characters
6. **ACL removal** when LoggedOut event fires without Node field

### 7.2 Implementation Best Practices (Recommended)

These align with advisory language in the specification:

1. **Aggressive timeouts** for PIN-to-TempId mappings (minimize accidental logins)
2. **Alphanumeric PINs** preferred over numeric-only (increased entropy)
3. **Unique TempAccountIdentifiers** per session (prevent correlation)
4. **Logging** of rate limit enforcement for security monitoring

### 7.3 Deployment Considerations

1. **Admin distribution**: Limit admin privilege to essential personnel
2. **Fabric isolation**: Ensure Content Apps are on appropriate fabrics
3. **External service security**: Secure the central account verification service
4. **ACL audit**: Periodically review ACL entries for stale accounts
