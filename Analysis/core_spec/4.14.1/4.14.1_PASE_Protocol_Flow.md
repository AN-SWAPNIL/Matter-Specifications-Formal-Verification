## Protocol Overview

PASE is a password-authenticated key exchange protocol used in Matter for initial device commissioning. It enables two parties (Commissioner and Commissionee) to establish a secure session using only a shared low-entropy passcode, providing:

- **Mutual Authentication**: Both parties verify each other's knowledge of the passcode
- **Forward Secrecy**: Session keys are ephemeral and independent
- **Resistance to Offline Dictionary Attacks**: SPAKE2+ protects against password guessing
- **Key Establishment**: Derives session encryption keys and attestation challenge

### Key Features

- Zero prior security context required
- Passcode can be 6-8 digit PIN or longer passphrase
- Protects against man-in-the-middle attacks
- Provides cryptographic proof of passcode knowledge without revealing it

## Roles and Terminology

### Parties

| Role          | Symbol | Description                                        |
| ------------- | ------ | -------------------------------------------------- |
| **Initiator** | I      | Commissioner (controller initiating commissioning) |
| **Responder** | R      | Commissionee (device being commissioned)           |

### Cryptographic Values

| Symbol       | Name              | Description                                     |
| ------------ | ----------------- | ----------------------------------------------- |
| **passcode** | Shared Secret     | Low-entropy password/PIN known to both parties  |
| **w0, w1**   | PAKE Values       | Derived from passcode via PBKDF                 |
| **L**        | Verifier          | w1 \* G (stored at responder)                   |
| **M, N**     | Blinding Points   | SPAKE2+ fixed elliptic curve points             |
| **G**        | Generator         | Base point of NIST P-256 curve                  |
| **pA**       | Initiator Public  | w0\*G + M (sent by initiator)                   |
| **pB**       | Responder Public  | w0\*G + N (sent by responder)                   |
| **Z, V**     | Shared Secrets    | ECDH-derived values from pA and pB              |
| **Ke**       | Encryption Key    | Derived from Z and V for session key derivation |
| **KcA, KcB** | Confirmation Keys | Used to generate cA and cB confirmations        |
| **cA, cB**   | Confirmations     | HMAC values proving passcode knowledge          |

### Session Keys (Final Output)

| Key                       | Symbol               | Purpose                               |
| ------------------------- | -------------------- | ------------------------------------- |
| **I2R Key**               | I2RKey               | Initiator-to-Responder encryption key |
| **R2I Key**               | R2IKey               | Responder-to-Initiator encryption key |
| **Attestation Challenge** | AttestationChallenge | Used in device attestation phase      |

## Cryptographic Primitives

### 1. PBKDF (Password-Based Key Derivation Function)

```
Input:  passcode, salt, iterations
Output: (w0, w1)

Purpose: Derive cryptographic values from low-entropy passcode
         Provides resistance to brute-force attacks
```

**Matter Parameters:**

- **Iterations**: 2000 (default) - configurable 1000-100000
- **Salt**: 16-32 bytes random value
- **Output**: 64 bytes (w0 = 32 bytes, w1 = 32 bytes)

### 2. SPAKE2+ Point Operations

```
Initiator computes:
  pA = w0 * G + M

Responder computes:
  pB = w0 * G + N
  L  = w1 * G  (verifier, computed once during setup)
```

**Fixed Points (Compressed Format):**

```
M = 02886e2f97ace46e55ba9dd7242579f2993b64e16ef3dcab95afd497333d8fa12f
N = 03d8bbd6c639c62937b04d997f38c3770719c629d7014d49a24b4f98baa1292b49
```

### 3. Shared Secret Computation

```
Z = ECDH((pB - w0*G - N), random_A)
  = Random component of shared secret

V = ECDH((pB - w0*G - N), w1)
  = Verifier component of shared secret
```

### 4. Transcript Formation (470 bytes total)

```
TT = Context || 0x00...00 || M || N || pA || pB || Z || V || w0

Where:
  Context      = hash(PBKDFParamRequest || PBKDFParamResponse)  [32 bytes]
  Padding      = 64 zero bytes
  M, N         = SPAKE2+ points                                 [65 bytes each]
  pA, pB       = Public points                                  [65 bytes each]
  Z, V         = Shared secrets                                 [65 bytes each]
  w0           = First PBKDF output                             [32 bytes]
```

### 5. Key Derivation

```
TT_hash = hash(TT)

KcA = HMAC-SHA256(TT_hash, "ConfirmationKeys" || 0x01)
KcB = HMAC-SHA256(TT_hash, "ConfirmationKeys" || 0x02)
Ke  = HMAC-SHA256(TT_hash, "SharedKey")

cA = HMAC-SHA256(KcA, pB)  [Initiator confirmation]
cB = HMAC-SHA256(KcB, pA)  [Responder confirmation]
```

### 6. Session Key Derivation

```
SessionKeys = KDF(Ke, "SessionKeys")

I2RKey              = SessionKeys[0:16]
R2IKey              = SessionKeys[16:32]
AttestationChallenge = SessionKeys[32:48]
```

## Message Flow

### Complete Exchange (8 Messages)

```
Initiator (I)                                    Responder (R)
═══════════════════════════════════════════════════════════════

1. Generate Random IR, Session ID
   |
   |---- PBKDFParamRequest ---->|
   |    (IR, SID, PasscodeID)   |
   |                            | 2. Validate PasscodeID=0
   |                            |    Generate Random RR, SID
   |                            |
   |<--- PBKDFParamResponse ----|
   |    (IR, RR, SID, Salt, I)  |
   |                            |
3. Verify IR echo               |
   Compute w0, w1 = PBKDF(...)  |
   Compute pA = w0*G + M        |
   |                            |
   |------ Pake1 (pA) -------->|
   |                            | 4. Compute w0, w1 = PBKDF(...)
   |                            |    Compute pB = w0*G + N
   |                            |    Compute Z, V, TT, Ke
   |                            |    Compute cB = HMAC(KcB, pA)
   |                            |
   |<--- Pake2 (pB, cB) --------|
   |                            |
5. Compute Z, V, TT, Ke         |
   Compute cB' = HMAC(KcB, pA)  |
   Verify cB' == cB ✓           |
   Compute cA = HMAC(KcA, pB)   |
   |                            |
   |------ Pake3 (cA) -------->|
   |                            | 6. Compute cA' = HMAC(KcA, pB)
   |                            |    Verify cA' == cA ✓
   |                            |    Derive Session Keys
   |                            |
   |<---- PakeFinished ---------|
   |                            |
7. Derive Session Keys          |
   Session Established ✓        | Session Established ✓
```

## Detailed Phase-by-Phase Execution

### **Phase 1: PBKDF Parameter Negotiation**

#### Message 1: PBKDFParamRequest (Initiator → Responder)

**Purpose:** Initiate PASE and request PBKDF parameters

**Initiator Actions:**

1. Generate fresh random value: `IR` (32 bytes)
2. Generate session ID: `InitiatorSessionID` (16-bit)
3. Set `PasscodeID = 0` (default passcode)
4. Set `hasPBKDFParameters = false` (request parameters)

**Message Structure:**

```
{
  initiatorRandom: IR,              // 32 bytes nonce
  initiatorSessionId: SID,          // 16-bit session identifier
  passcodeId: 0,                    // Default passcode
  hasPBKDFParameters: false         // Request PBKDF params
}
```

**Security Checks:**

- IR MUST be cryptographically random
- Session ID MUST be unique for this node
- PasscodeID=0 is the only valid value for commissioning

---

#### Message 2: PBKDFParamResponse (Responder → Initiator)

**Purpose:** Echo initiator random and provide PBKDF parameters

**Responder Actions:**

1. **Validate** `PasscodeID == 0`
2. Generate fresh random: `RR` (32 bytes)
3. Generate session ID: `ResponderSessionID`
4. Retrieve or generate PBKDF salt (16-32 bytes)
5. Set iteration count (default: 2000)

**Message Structure:**

```
{
  initiatorRandomEcho: IR,          // Echo for replay protection
  responderRandom: RR,              // 32 bytes fresh nonce
  responderSessionId: RSID,         // 16-bit session identifier
  pbkdfSalt: Salt,                  // 16-32 bytes
  pbkdfIterations: 2000             // Iteration count
}
```

**Security Checks:**

- RR MUST be cryptographically random
- Salt should be device-unique and persistent
- Iterations in range [1000, 100000]

**Initiator Validation:**

- Verify `initiatorRandomEcho == IR`
- Abort if mismatch (replay attack detected)

---

### **Phase 2: SPAKE2+ Public Point Exchange**

#### Pre-Message Computation (Both Parties)

**Derive PAKE Values:**

```
pakeVals = PBKDF(passcode, salt, iterations)
w0 = pakeVals[0:32]   // First 32 bytes
w1 = pakeVals[32:64]  // Second 32 bytes
```

**Responder Pre-Computes Verifier:**

```
L = w1 * G  // Can be computed once and stored
```

---

#### Message 3: Pake1 (Initiator → Responder)

**Purpose:** Send initiator's SPAKE2+ public point

**Initiator Actions:**

1. Generate ephemeral scalar `x` (random)
2. Compute `pA = x*G + w0*G + M`
   - Simplified: `pA = (x + w0)*G + M`

**Message Structure:**

```
{
  pA: Point  // 65 bytes (uncompressed) or 33 bytes (compressed)
}
```

**Security Properties:**

- `x` MUST be random and ephemeral
- `pA` hides `w0` due to blinding with `M`
- Even if attacker knows `M` and sees `pA`, cannot extract `w0`

---

#### Message 4: Pake2 (Responder → Initiator)

**Purpose:** Send responder's public point and confirmation

**Responder Actions:**

1. Generate ephemeral scalar `y` (random)
2. Compute `pB = (y + w0)*G + N`
3. Compute shared secrets:
   ```
   Z = y * (pA - w0*G - M)  // ECDH with initiator's random component
   V = y * L = y * w1 * G   // ECDH with verifier
   ```
4. Compute transcript:
   ```
   Context = hash(PBKDFParamRequest || PBKDFParamResponse)
   TT = Context || padding || M || N || pA || pB || Z || V || w0
   TT_hash = hash(TT)
   ```
5. Derive keys:
   ```
   KcA = derive_KcA(TT_hash)
   KcB = derive_KcB(TT_hash)
   Ke  = derive_Ke(TT_hash)
   ```
6. Compute confirmation:
   ```
   cB = HMAC(KcB, pA)
   ```

**Message Structure:**

```
{
  pB: Point,           // 65 bytes
  cB: Confirmation     // 32 bytes HMAC
}
```

**Security Properties:**

- `cB` proves responder knows passcode
- Cannot be forged without knowing `w0` and `w1`
- Binds to specific `pA` value

---

### **Phase 3: Mutual Authentication**

#### Initiator Verification (After Receiving Pake2)

**Initiator Actions:**

1. Compute shared secrets:

   ```
   Z = x * (pB - w0*G - N)
   V = x * L = x * w1 * G
   ```

   Note: Initiator computes same Z, V as responder due to ECDH symmetry

2. Compute same transcript:

   ```
   Context = hash(PBKDFParamRequest || PBKDFParamResponse)
   TT = Context || padding || M || N || pA || pB || Z || V || w0
   TT_hash = hash(TT)
   ```

3. Derive same keys:

   ```
   KcA = derive_KcA(TT_hash)
   KcB = derive_KcB(TT_hash)
   Ke  = derive_Ke(TT_hash)
   ```

4. **VERIFY cB:**

   ```
   cB_computed = HMAC(KcB, pA)
   if (cB_received != cB_computed) {
       ABORT // Responder doesn't know correct passcode
   }
   ```

5. If verification succeeds, compute initiator confirmation:
   ```
   cA = HMAC(KcA, pB)
   ```

**Critical Security Check:**

- **MUST NOT** derive session keys before `cB` verification
- Failure indicates wrong passcode or attack

---

#### Message 5: Pake3 (Initiator → Responder)

**Purpose:** Prove initiator knows passcode

**Message Structure:**

```
{
  cA: Confirmation  // 32 bytes HMAC
}
```

**Initiator State After Sending:**

- Waiting for PakeFinished acknowledgment
- Ready to derive session keys upon success

---

#### Responder Verification (After Receiving Pake3)

**Responder Actions:**

1. Compute expected confirmation:

   ```
   cA_computed = HMAC(KcA, pB)
   ```

2. **VERIFY cA:**

   ```
   if (cA_received != cA_computed) {
       ABORT // Initiator doesn't know correct passcode
   }
   ```

3. If verification succeeds, derive session keys:

   ```
   SessionKeys = KDF(Ke, "SessionKeys")
   I2RKey = SessionKeys[0:16]
   R2IKey = SessionKeys[16:32]
   AttestationChallenge = SessionKeys[32:48]
   ```

4. Send PakeFinished

---

#### Message 6: PakeFinished (Responder → Initiator)

**Purpose:** Acknowledge successful mutual authentication

**Message Structure:**

```
{
  status: SUCCESS
}
```

**Responder State After Sending:**

- Session keys active
- Ready for encrypted communication
- Can process attestation with AttestationChallenge

---

### **Phase 4: Session Key Derivation**

**Initiator Actions (After Receiving PakeFinished):**

1. Derive session keys:

   ```
   SessionKeys = KDF(Ke, "SessionKeys")
   I2RKey = SessionKeys[0:16]
   R2IKey = SessionKeys[16:32]
   AttestationChallenge = SessionKeys[32:48]
   ```

2. Mark session as established
3. Begin encrypted messaging using I2RKey/R2IKey

**Session Established Successfully**

Both parties now share:

- **I2RKey**: For initiator→responder encryption (AES-128-CCM)
- **R2IKey**: For responder→initiator encryption (AES-128-CCM)
- **AttestationChallenge**: Used in subsequent device attestation

## Security Properties

### SP1: Mutual Authentication

**Property:** Both parties must verify confirmation values  
**Guarantee:** Session established only if both know correct passcode  
**Implementation:** cB verification at initiator, cA verification at responder

### SP2: Key Secrecy

**Property:** Session keys remain secret to outsiders  
**Guarantee:** Attacker without passcode cannot learn I2RKey or R2IKey  
**Verified:** ProVerif confirms keys cannot be derived by attacker

### SP3: Forward Secrecy

**Property:** Session keys independent across sessions  
**Guarantee:** Compromise of one session doesn't affect others  
**Implementation:** Fresh random values (IR, RR, x, y) each session

### SP4: Offline Dictionary Attack Resistance

**Property:** Attacker cannot brute-force passcode offline  
**Guarantee:** SPAKE2+ requires online interaction for each guess  
**Implementation:** Blinding with M/N points, confirmation values

### SP5: Replay Protection

**Property:** Cannot replay old messages  
**Guarantee:** Fresh randoms (IR, RR) and session IDs  
**Verification:** IR echo in PBKDFParamResponse

### SP6: Integrity

**Property:** Both parties agree on pA and pB  
**Guarantee:** Transcript includes all public values  
**Implementation:** TT hash binds all protocol messages

### SP7: No Premature Encryption

**Property:** Keys derived only after authentication  
**Guarantee:** cB/cA verified before session key derivation  
**Verified:** ProVerif confirms order of events

### SP8: Attestation Challenge Secrecy

**Property:** AttestationChallenge known only to session parties  
**Use Case:** Prevents attestation bypass attacks

## Failure Scenarios

### F1: Wrong Passcode (Initiator Detects)

```
Initiator receives Pake2 with invalid cB
→ cB_computed ≠ cB_received
→ Event: Verify_cB_Failure
→ Action: Abort, send failure status
→ Result: Session NOT established
```

### F2: Wrong Passcode (Responder Detects)

```
Responder receives Pake3 with invalid cA
→ cA_computed ≠ cA_received
→ Event: Verify_cA_Failure
→ Action: Abort, send failure status
→ Result: Session NOT established
```

### F3: Invalid PasscodeID

```
Responder receives PBKDFParamRequest with PasscodeID ≠ 0
→ Event: InvalidPasscodeId
→ Action: Reject immediately
→ Result: No PBKDF parameter exchange
```

### F4: Replay Attack

```
Initiator receives PBKDFParamResponse
→ initiatorRandomEcho ≠ IR
→ Action: Abort
→ Result: Session NOT established
```

### F5: Timeout

```
Any party waiting for message > timeout threshold
→ Action: Abort current session
→ Result: Session NOT established
→ Recovery: Retry from Phase 1
```

### F6: Computational Error

```
Invalid elliptic curve point received
→ Point validation fails
→ Action: Abort
→ Result: Session NOT established
```

## State Transitions

### Initiator State Machine

```
[IDLE]
  ↓ (User initiates commissioning)
[T1: Generating_IR]
  ↓ (Generate IR, SessionID)
[T2: Sent_PBKDFParamRequest]
  ↓ (Send PBKDFParamRequest)
[T3: Waiting_PBKDFParamResponse]
  ↓ (Receive PBKDFParamResponse)
[Computing_PBKDF]
  ↓ (Compute w0, w1)
[Computing_pA]
  ↓ (Compute pA)
[T4: Sent_Pake1]
  ↓ (Send Pake1)
[T5: Waiting_Pake2]
  ↓ (Receive Pake2)
[Verifying_cB]
  ├─ (cB valid) → [Computing_cA]
  └─ (cB invalid) → [Failed_Authentication]
[T6: Sent_Pake3]
  ↓ (Send Pake3)
[T7: Waiting_PakeFinished]
  ↓ (Receive PakeFinished)
[Deriving_SessionKeys]
  ↓ (Compute I2RKey, R2IKey, AttestationChallenge)
[SessionEstablished]
```

### Responder State Machine

```
[IDLE]
  ↓ (Listen for commissioning)
[T1: Waiting_PBKDFParamRequest]
  ↓ (Receive PBKDFParamRequest)
[Validating_PasscodeID]
  ├─ (Valid) → [Generating_RR]
  └─ (Invalid) → [Failed_InvalidPasscodeID]
[T2: Sent_PBKDFParamResponse]
  ↓ (Send PBKDFParamResponse)
[T3: Waiting_Pake1]
  ↓ (Receive Pake1)
[Computing_PBKDF]
  ↓ (Compute w0, w1)
[Computing_SharedSecrets]
  ↓ (Compute Z, V, Ke)
[Computing_cB]
  ↓ (Compute cB)
[T4: Sent_Pake2]
  ↓ (Send Pake2)
[T5: Waiting_Pake3]
  ↓ (Receive Pake3)
[Verifying_cA]
  ├─ (cA valid) → [Deriving_SessionKeys]
  └─ (cA invalid) → [Failed_Authentication]
[T6: Sent_PakeFinished]
  ↓ (Send PakeFinished)
[SessionEstablished]
```

## Implementation Notes

### Timing Considerations

- PBKDF computation: ~100ms (device-dependent)
- Total protocol time: <500ms on typical hardware
- Timeout recommendation: 30 seconds per phase

### Nonce Construction (13 bytes)

```
Nonce = NodeID (8 bytes) || MessageCounter (4 bytes) || 0x00
```

### Session ID Management

- 16-bit value (0-65535)
- MUST be unique per active session
- Collision handling: reject duplicate session IDs

### Message Header Validation

```
Security Flags:
  P (Privacy): MUST be 0 for PASE
  C (Control): 0
  MX (Message Extension): 0
  Reserved: 0
  Session Type: 0 (Unsecured)

Message Flags:
  Version: 0
  S (Source Node ID): 1 (present)
  DSIZ: 00 (Destination omitted)
```

### Backwards Compatibility

- Matter 1.3+: All 9 session parameters supported
- Pre-1.3: Only tags 1-7 required
- Default values for missing optional parameters
