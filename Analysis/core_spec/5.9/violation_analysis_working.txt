POTENTIAL VIOLATION 1: Timing Window Attack
Property: PROP_017 (Subsequent_Messages_Over_Operational_Network)
Issue: Between OperationalConnected and OperationalDiscovered, is recovery connection truly terminated?

FSM shows:
- ConnectingToOperational -> OperationalConnected
  Action: "terminate_recovery_connection()"
  
- OperationalConnected -> OperationalDiscovered
  Trigger: "initiate_operational_discovery"
  
- OperationalDiscovered -> RecoveryCompleted
  Trigger: "commissioning_complete_command"
  Guard: "received_over_operational_network == true && over_case_session == true"

But what about messages BEFORE CommissioningComplete?
- If credential provisioning or ConnectNetwork messages come after operational connection but before discovery?
- FSM doesn't show explicit transitions for those

Actually, credential provisioning happens BEFORE operational connection.
The concern is: Can CommissioningComplete be sent over recovery channel?
The guard explicitly checks "received_over_operational_network == true"
So this is enforced.

NOT A VIOLATION.

POTENTIAL VIOLATION 2: Recovery Connection Not Truly Terminated
Property: PROP_033 (Recovery_Connection_Terminated_After_Operational_Connection)

Issue: terminate_recovery_connection() is called, but there's no guard preventing use of recovery connection after this point.

Between OperationalConnected and RecoveryCompleted:
- Could Administrator still send messages over recovery channel (BLE/WiFi)?
- FSM doesn't model individual message sends
- Only CommissioningComplete has guard checking channel

If Administrator sends OTHER commands (e.g., read attributes, configure settings) over recovery channel after OperationalConnected, would they be accepted?

FSM shows:
- OperationalConnected state requires "recovery_connection_terminated == true"
- But there's no transition showing rejection of recovery channel messages
- Only CommissioningComplete has explicit channel check

This is a GAP in FSM modeling, not necessarily a violation.
The specification says recovery CASE "SHALL be used for recovery and not for steady state"
But FSM doesn't model enforcement of this after connection.

VERDICT: PARTIAL VIOLATION - FSM doesn't enforce recovery channel inactivity post-connection for non-CommissioningComplete messages.

Actually, looking at PROP_025:
"Recovery Nodes SHALL allow any and all transactions allowed over a CASE session"
This applies DURING recovery, from Step 9.

And PROP_028 says:
"CASE session established over commissioning channel during recovery must be used only for network credential recovery"

So there's a contradiction:
- PROP_025 says allow all CASE transactions during recovery
- PROP_028 says use only for recovery, not operational traffic

Hmm, but PROP_025 talks about "during recovery" (Steps 9-16)
And PROP_028 talks about "not for steady-state operational interaction"

The key is: what's "steady-state operational" vs "recovery-related configuration"?

Actually, re-reading Step 9:
"Recovery Nodes SHALL allow any and all transactions allowed over a CASE session, such as reading Basic Information cluster or accessing device configuration data and topology, as Administrators might require them to fully restore network connectivity."

VIOLATION 1: PROP_021 - Continuous_Reconnection_During_Recovery_State
----------------------------------------------------------------------

Property Claim:
"Recovery Nodes must attempt to reconnect to existing operational network while in the recovery state."

6. RecoveryConnectionEstablished -> CASESessionEstablished: suspension happens HERE

Gap Period: From AwaitingAdministrator through RecoveryConnectionEstablished (before CASE established)
- Node is still in "recovery state" (recovery_mode == true)
- But no reconnection attempts modeled in FSM
- Violates continuous reconnection requirement

Specification Evidence:
Quote: "Recovery Nodes SHALL attempt to reconnect to the existing operational network while in the recovery state."
Source: Section 5.9.3, Step 3

Additional Support:
Quote: "once the secure connection is established, any ongoing attempts taken autonomously by the Recovery Node to re-establish connectivity to the previously configured operational network SHALL be suspended"
Source: Section 5.9.3, Step 9
Context: This implies reconnection attempts are ONGOING until CASE session, but FSM doesn't model this between RecoveryAnnouncing and CASESessionEstablished

The FSM only models reconnection attempts in:
- WaitingReconnect (stay transition)
- RecoveryAnnouncing (stay transition)
- ExtendedAnnouncementEnded (stay transition)

Missing reconnection in:
- AwaitingAdministrator
- RecoveryConnectionEstablished

Severity: MEDIUM (creates window where node stops attempting autonomous recovery)

Impact:
- If network becomes available during AwaitingAdministrator or RecoveryConnectionEstablished states
- Node will NOT autonomously reconnect
- Forces dependency on Administrator-assisted recovery even when autonomous recovery could succeed
- Increases recovery time unnecessarily
